# NestJS CLI и Структура проекта

NestJS предоставляет мощный инструмент командной строки (CLI), который берет на себя рутину по созданию файлов, настройке boilerplate-кода и сборке проекта. Кроме того, фреймворк формирует четкую и интуитивно понятную структуру папок, что упрощает навигацию по мере масштабирования кодовой базы.

---

## Инструмент Nest CLI

**Nest CLI** — это консольная утилита, которая помогает инициализировать, разрабатывать и поддерживать приложения Nest.

### Установка и базовые команды

Для начала убедитесь, что CLI установлен глобально:
```bash
npm install -g @nestjs/cli
```

*   `nest new <project-name>` — создает новый проект со стандартной структурой и устанавливает необходимые зависимости.
*   `nest start` — запускает приложение. Поддерживает флаг `--watch` (или `npm run start:dev`) для автоматической перезагрузки при изменении исходного кода.
*   `nest build` — компилирует TypeScript-код в JavaScript (сохраняется в папку `dist`).
*   `nest generate <schematic> <name>` (или просто `nest g`) — удобная команда для генерации компонентов:
    *   `nest g module users` — создает папку `users` с файлом `users.module.ts`.
    *   `nest g controller users` / `nest g service users` — генерирует контроллеры и сервисы, автоматически добавляя их в соответствующий модуль.
    *   `nest g resource users` — мощная команда, генерирующая полноценный CRUD-модуль со всеми слоями (контроллер, сервис, DTO, сущности/entities).

## Структура проекта изнутри

После создания нового проекта (командой `nest new`) в директории `src/` генерируется базовая структура приложения:

```text
src/
├── app.controller.spec.ts   # Unit-тесты для контроллера
├── app.controller.ts        # Базовый контроллер (маршрутизатор)
├── app.module.ts            # Корневой модуль приложения
├── app.service.ts           # Базовый сервис (бизнес-логика)
└── main.ts                  # Точка входа в приложение
```

### Разбор ключевых файлов:

1.  **`main.ts`**
    Здесь используется класс `NestFactory` для создания экземпляра приложения на основе корневого модуля (`AppModule`). В этом файле также настраивается порт, назначаются глобальные `Pipes`, префиксы API и CORS.
    ```typescript
    import { NestFactory } from '@nestjs/core';
    import { AppModule } from './app.module';

    async function bootstrap() {
      const app = await NestFactory.create(AppModule);
      await app.listen(3000); // Запуск сервера на порту 3000
    }
    bootstrap();
    ```

2.  **`app.module.ts`**
    Корневой модуль (`@Module()`), собирающий всё приложение воедино. По мере роста проекта сюда импортируются все остальные feature-модули (например, `UsersModule`, `AuthModule`, `ProductModule`).

3.  **`app.controller.ts`**
    Контроллер, отвечающий за приём входящих HTTP-запросов и возврат ответов клиенту.

4.  **`app.service.ts`**
    Пример простого провайдера (сервиса). Сервисы выполняют бизнес-логику и могут переиспользоваться («внедряться») в контроллеры или другие сервисы благодаря встроенному Dependency Injection.

## Важные конфигурационные файлы (в корне проекта)

Помимо папки `src`, в корне сгенерированного проекта можно найти важные файлы для TypeScript/Node.js разработки:

*   **`nest-cli.json`**: Конфигурация для Nest CLI. Управляет сборкой, путями к исходникам и подключением плагинов (например, для автоматической генерации схем Swagger или GraphQL).
*   **`tsconfig.json`** / **`tsconfig.build.json`**: Настройки компилятора TypeScript (по умолчанию используется строгий режим (strict mode)).
*   **`package.json`**: Содержит список зависимостей и заранее настроенные скрипты: `start:dev`, `build`, `lint`, `test`, `test:e2e`.
*   **`test/`**: Папка для End-to-End (e2e) тестирования. Фреймворк «из коробки» настроен на использование **Jest** для тестов и **Supertest** для эмуляции и проверки HTTP-запросов.

## Итог

Структура проекта в NestJS спроектирована так, чтобы с первого дня навязывать надежное разделение ответственности (Separation of Concerns): 
* **Контроллеры** — только принимают и отдают данные;
* **Сервисы** — инкапсулируют бизнес-логику;
* **Модули** — группируют связанные компоненты.

В следующем материале мы более детально разберем концепцию **Модулей (Modules)** и узнаем, как грамотно разбивать приложение на слабо связные домены.
