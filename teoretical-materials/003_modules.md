# Модули в NestJS (Modules)

В NestJS приложение строится как дерево модулей. **Модуль** — это класс, аннотированный декоратором `@Module()`. Декоратор предоставляет метаданные, которые фреймворк использует для организации структуры приложения и разрешения зависимостей (Dependency Injection).

У каждого приложения есть как минимум один модуль — **Корневой модуль (Root Module)**, обычно называемый `AppModule`. Он служит отправной точкой, с которой NestJS начинает строить граф зависимостей.

---

## Зачем нужны модули?

В отличие от Express, где файлы собираются воедино вручную, NestJS требует явного объявления связей. Модули позволяют:
1.  **Группировать логику по фичам (Feature Modules):** Например, все контроллеры, сервисы и сущности, относящиеся к `Users`, находятся в одном `UsersModule`.
2.  **Инкапсулировать зависимости:** По умолчанию провайдеры (сервисы) спрятаны внутри своего модуля. Они не доступны извне, пока не будут явно экспортированы.
3.  **Переиспользовать код (Shared Modules):** Создав модуль базы данных или логгирования, вы можете импортировать его в любые другие модули приложения.

---

## Структура декоратора `@Module()`

Декоратор `@Module()` принимает объект со следующими свойствами:

```typescript
import { Module } from '@nestjs/common';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  imports: [],
  controllers: [UsersController],
  providers: [UsersService],
  exports: [UsersService]
})
export class UsersModule {}
```

### Разбор свойств (Properties):

*   **`controllers`**: Массив контроллеров, которые определены в этом модуле. NestJS автоматически создаст им маршруты и будет обрабатывать входящие HTTP-запросы.
*   **`providers`**: Массив провайдеров (сервисы, репозитории, хелперы), которые будут инстанцированы контейнером `Nest (IoC)` и могут быть внедрены (injected) внутри этого модуля.
*   **`imports`**: Массив других модулей, которые требуются для работы текущего. Если `UsersModule` использует сервис из `AuthModule`, то `AuthModule` должен быть добавлен в этот массив.
*   **`exports`**: Массив провайдеров (из массива `providers` этого же модуля), которые должны быть доступны в других модулях, импортирующих текущий. Это механизм создания публичного API (интерфейса) модуля.

## Feature Modules (Модули функциональности)

Хорошей практикой (и стандартом) является организация кода по предметным областям (Domain Driven Design). Если вы делаете систему управления пользователями, создайте `UsersModule`. 

**Как сгенерировать модуль через CLI:**
```bash
nest g module users
```

Эта команда создаст папку `users/` с файлом `users.module.ts` и автоматически добавит импорт `UsersModule` в корневой `AppModule`.

## Shared Modules (Глобальные и общие модули)

В NestJS модули по умолчанию являются синглтонами (singleton). Это значит, что если переиспользуемый `DatabaseModule` импортирован в нескольких местах, это всегда будет один и тот же экземпляр с одним и тем же пулом подключений.

### Глобальные модули (`@Global()`)
Если у вас есть провайдеры, которые нужны *везде* (например, подключение к БД или сервис конфигурации), их можно сделать глобальными, чтобы не прописывать импорт в каждом отдельном модуле.

```typescript
import { Module, Global } from '@nestjs/common';
import { DatabaseService } from './database.service';

@Global()
@Module({
  providers: [DatabaseService],
  exports: [DatabaseService],
})
export class DatabaseModule {}
```
*Примечание: злоупотреблять `@Global()` не стоит. Модули должны явно декларировать свои зависимости для лучшей читаемости и надежности кода.*

## Итог

Модули — это контейнеры, формирующие архитектуру приложения. Они строго изолируют функционал и управляют видимостью зависимостей (через `imports` и `exports`).

В следующих разделах мы разберем основные "кирпичики", из которых состоят модули: **Контроллеры (возня с маршрутами)** и **Провайдеры (где живет вся бизнес-логика)**.
