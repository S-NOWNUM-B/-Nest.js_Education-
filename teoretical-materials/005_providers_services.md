# Провайдеры и Сервисы (Providers & Services)

**Провайдер (Provider)** — это фундаментальная концепция в NestJS. Большинство базовых классов Nest могут рассматриваться как провайдеры: сервисы, репозитории, фабрики, хелперы и так далее. 

Главная идея провайдера заключается в том, что его можно **внедрить (inject)** в качестве зависимости в другие классы (например, в контроллеры или другие сервисы). Благодаря этому классы не создают свои зависимости руками (через `new`), а получают их от встроенного **IoC-контейнера (Inversion of Control)**.

---

## Что такое Service?

**Сервис (Service)** — это самая распространенная форма Провайдера. Если Контроллеры нужны только для того, чтобы принимать HTTP-запросы и отдавать ответы, то Сервисы создаются для выполнения всей сложной, грязной работы — **бизнес-логики**.

Сервисы отвечают за:
* Поиск, сохранение и удаление данных в базе данных.
* Формирование бизнес-сущностей, расчеты, вызовы сторонних API.
* Реализацию алгоритмов, которые Контроллеру знать не нужно.

---

## Создание Сервиса

Создать сервис можно вручную, но проще через Nest CLI:
```bash
nest g service users
```
Эта команда создаст файл `users.service.ts` и добавит его в массив `providers` в соответствующем модуле.

### Базовый пример Сервиса

Чтобы обычный класс стал "Провайдером", который понимает NestJS, перед ним необходимо добавить декоратор `@Injectable()`.

```typescript
// users.service.ts
import { Injectable } from '@nestjs/common';

// Интерфейс для типизации (в реальности обычно хранится в отдельном файле)
export interface User {
  id: number;
  name: string;
}

@Injectable() // Этот декоратор делает класс Провайдером
export class UsersService {
  // Имитация базы данных
  private readonly users: User[] = [];

  create(user: User) {
    this.users.push(user);
  }

  findAll(): User[] {
    return this.users;
  }
}
```

---

## Внедрение Провайдера в Контроллер (Dependency Injection)

Давайте посмотрим, как использовать созданный сервис внутри `UsersController`. Для этого используется **Dependency Injection (DI) через конструктор**.

Мы просто прописываем тип нужного нам сервиса в параметрах конструктора Контроллера, и NestJS сам инстанцирует этот сервис и передаст его при создании класса.

```typescript
// users.controller.ts
import { Controller, Get, Post, Body } from '@nestjs/common';
import { UsersService, User } from './users.service';

@Controller('users')
export class UsersController {
  
  // Внедрение зависимости!
  // Ключевое слово 'private readonly' автоматически создает свойство класса 'usersService'
  constructor(private readonly usersService: UsersService) {}

  @Post()
  async create(@Body() user: User) {
    // Вся бизнес логика делегирована в Сервис
    this.usersService.create(user); 
    return 'Пользователь создан!';
  }

  @Get()
  async findAll(): Promise<User[]> {
    return this.usersService.findAll();
  }
}
```

## Регистрация Провайдера в Модуле

Чрезвычайно важно понимать: чтобы Контроллер мог "попросить" `UsersService`, этот сервис должен быть зарегистрирован в Модуле, к которому привязан Контроллер.

Происходит это в файле `.module.ts` через массив `providers`:

```typescript
// users.module.ts
import { Module } from '@nestjs/common';
import { UsersController } from './users.controller';
import { UsersService } from './users.service';

@Module({
  controllers: [UsersController],
  providers: [UsersService], // Регистрация Провайдера в контейнере!
})
export class UsersModule {}
```

Если вы забудете указать класс в массиве `providers`, при запуске приложения Nest выдаст ошибку, сообщающую о том, что он "не может разрешить зависимости".

---

## Итог

*   **Контроллер** отвечает за связи (HTTP) и делегирует работу.
*   **Сервис (Провайдер)** содержит логику (`@Injectable()`).
*   Связывание происходит автоматически благодаря **внедрению через конструктор** (Dependency Injection).
*   Всё это работает, только если Провайдер зарегистрирован в **Модуле** (`providers: [...]`).
